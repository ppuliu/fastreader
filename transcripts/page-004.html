5	## When to Use
6	
7	Decide per-question, not per-session. The test: **would the user understand this better by seeing it than reading it?**
8	
9	**Use the browser** when the content itself is visual:
10	
11	- **UI mockups** — wireframes, layouts, navigation structures, component designs
12	- **Architecture diagrams** — system components, data flow, relationship maps
13	- **Side-by-side visual comparisons** — comparing two layouts, two color schemes, two design directions
14	- **Design polish** — when the question is about look and feel, spacing, visual hierarchy
15	- **Spatial relationships** — state machines, flowcharts, entity relationships rendered as diagrams
16	
17	**Use the terminal** when the content is text or tabular:
18	
19	- **Requirements and scope questions** — "what does X mean?", "which features are in scope?"
20	- **Conceptual A/B/C choices** — picking between approaches described in words
21	- **Tradeoff lists** — pros/cons, comparison tables
22	- **Technical decisions** — API design, data modeling, architectural approach selection
23	- **Clarifying questions** — anything where the answer is words, not a visual preference
24	
25	A question *about* a UI topic is not automatically a visual question. "What kind of wizard do you want?" is conceptual — use the terminal. "Which of these wizard layouts feels right?" is visual — use the browser.

27	## How It Works
28	
29	The server watches a directory for HTML files and serves the newest one to the browser. You write HTML content to `screen_dir`, the user sees it in their browser and can click to select options. Selections are recorded to `state_dir/events` that you read on your next turn.

31	**Content fragments vs full documents:** If your HTML file starts with `<!DOCTYPE` or `<html`, the server serves it as-is (just injects the helper script). 99	
100	1. **Check server is alive**, then **write HTML** to a new file in `screen_dir`:
101	   - **Required: confirm the server is alive before referring to the URL or pushing a screen.** Check that `$STATE_DIR/server-info` exists and `$STATE_DIR/server-stopped` does not. 102	   - Use semantic filenames: `platform.html`, `visual-style.html`, `layout.html`
103	   - **Never reuse filenames** — each screen gets a fresh file
104	   - Use your file-creation tool — **never use cat/heredoc** (dumps noise into terminal)
105	   - Server automatically serves the newest file
106	
107	2. **Tell user what to expect and end your turn:**
108	   - Remind them of the URL (every step, not just first)
109	   - Give a brief text summary of what's on screen (e.g., "Showing 3 layout options for the homepage")
110	   - Ask them to respond in the terminal: "Take a look and let me know what you think. Click to select an option if you'd like."
111	
112	3. **On your next turn** — after the user responds in the terminal:
113	   - Read `$STATE_DIR/events` if it exists — this contains the user's browser interactions (clicks, selections) as JSON lines
114	   - Merge with the user's terminal text to get the full picture
115	   - The terminal message is the primary feedback; `state_dir/events` provides structured interaction data
116	
117	4. **Iterate or advance** — if feedback changes current screen, write a new file (e.g., `layout-v2.html`). Only move to the next question when the current step is validated.
118	
119	5. **Unload when returning to terminal** — when the next step doesn't need the browser (e.g., a clarifying question, a tradeoff discussion), push a waiting screen to clear the stale content:
120	
121	   ```html

124	     <p class="subtitle">Continuing in terminal...</p>

251	
252	When the user clicks options in the browser, their interactions are recorded to `$STATE_DIR/events` (one JSON object per line). The file is cleared automatically when you push a new screen.
253	
254	```jsonl
255	{"type":"click","choice":"a","text":"Option A - Simple Layout","timestamp":1706000101}

An Orca worktree/workspace is Orca's tracked view of a repo checkout, its metadata, terminals, browser tabs, and UI state.

Common commands:

orca worktree set --worktree active --comment "reproduced bug; testing fix" --json
orca worktree rm --worktree id:<worktreeId> --force --json

Selectors:

  • id:<worktreeId>, path:<absolutePath>, branch:<branchName>, issue:<number>
  • active / current for the enclosing Orca-managed worktree from the shell cwd

Lineage rules:

  • When creating from inside an Orca-managed worktree, Orca infers the current workspace as the parent when it can.
  • Use --parent-worktree active when the child relationship should be explicit.
  • Use --no-parent only when the new work is independent.
  • If --repo is omitted, Orca infers the repo from the current Orca worktree when possible.

Agent/setup flags:

orca worktree create --name task --agent claude --setup run --json
orca worktree create --name task --setup skip --json
orca worktree create --name task --run-hooks --json

  • --setup run|skip|inherit controls repo setup hooks. Default is inherit, which follows the repo's setup policy.

Automations

An automation is a scheduled Orca prompt run by a chosen provider against either a repo-created worktree or an existing workspace.

Level structure & anchoring semantics

How a document's levels get decided. Two hard rules and one heuristic. Rules: level 0 is always a single paragraph (the gist), and the last level is always the verbatim full text. Heuristic: each level should be 3–6× the word count of the one above, and intermediate levels snap to the document's natural structure where one exists (chapters for Alice, sections for the paper) rather than arbitrary math. So Alice gets 5 levels (~80 → ~450 → ~1.4k on chapter boundaries → ~4k → 26k) and the paper gets 4 (~100 → ~600 → ~2.5k on section boundaries → ~7k). A 300-page novel would get 6. The level count, names, and word counts all live in the document JSON; the frontend renders whatever it finds.

Which segment is "under your cursor." When zoom comes from pinch or ⌘-scroll, the anchor is the pointer's y-position — literally Google Maps semantics: the text under your cursor stays put. When it comes from the dial, keyboard, or a click on a dial stop, the anchor is the viewport center (like Maps' +/− buttons). We find the segment whose box contains that y, and that's the focal segment.

Where you land. Zooming out is unambiguous: focal segment → its parent, positioned so the parent sits at the anchor y. Zooming in has a nice subtlety: the focal segment maps to several children, so we pick proportionally — if your anchor was 70% of the way down the chapter paragraph, you land 70% of the way through its children (child index ≈ fraction × n). This makes repeated zoom-in feel like continuously diving toward the sentence you were reading, not repeatedly snapping to the top of each chapter.

Guard rails. Zoom during an in-flight transition is ignored (300ms window, keeps state simple for the prototype). Ends clamp — zooming out at gist and in at full text do a tiny rubber-band scale bounce instead of nothing, which is a cheap way to say "you've hit the edge of the world." Headings render inline at levels where segments align with named structure (chapter titles at the chapters level and below), and each segment's heading is optional in the data.

One honest simplification to flag: pinch zoom will be discrete (each gesture crossing a threshold = one level jump), not continuous scaling like real Maps. Continuous would mean rendering two levels at intermediate scales simultaneously — beautiful, but it's an afternoon by itself. The rubber-band bounce at the ends and the directional scale in the transition fake most of the continuous feel.

Does this section look right — anything you'd change about the anchoring rules or level heuristic before we move to architecture (data model + system diagram, back in the browser)?

Good catch — and shift+scroll is workable, with one OS-level gotcha worth knowing: macOS (and most mice drivers) convert shift+wheel into horizontal scroll, so the event arrives as deltaX, not deltaY. Since our reader has no horizontal scrolling at all, that's actually unambiguous — shiftKey && deltaX simply means "zoom" — but it's why shift+scroll alone would feel flaky if implemented naïvely.

My proposal is to be generous rather than precious about the modifier — any modifier + scroll = zoom, plain scroll = move through the document:

  • Plain scroll / trackpad two-finger — pan up and down within the level. Never hijacked.
  • Pinch — zoom (trackpads report pinch as a ctrlKey wheel event, so it's free).
  • Ctrl / ⌘ / Shift / Alt + scroll — all zoom. One wheel handler checks "any modifier held?" — Ctrl and ⌘ need preventDefault so the browser doesn't page-zoom (same interception Maps and Figma do), Shift needs the deltaX reading. Nobody has to remember which modifier; whatever they reach for works.
